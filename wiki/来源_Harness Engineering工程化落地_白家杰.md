# 来源：万字干货！Harness Engineering 如何工程化落地？

## 基本信息

- **作者**：[[白家杰]]
- **发布**：微信公众号，2026-04-23
- **来源链接**：https://mp.weixin.qq.com/s?__biz=MzI2NDU4OTExOQ==&mid=2247695342
- **案例项目**：JK Launcher（面向 Unity 项目研发流程的 Windows 桌面启动器，WPF/.NET Framework）
- **背景**：全文 12 章，作者以 JK Launcher 为真实样本，记录从零搭建 [[Harness Engineering]] 的完整演化过程

## 核心要点

1. **Harness 六大零件**：Rule（规矩）/ Skill（操作手册）/ Sub Agent（角色分工）/ Workflow（接力赛规则）/ Scripts（硬门禁）/ MCP（外接能力接口）——逐层叠加，各解决不同问题
2. **SPEC 先行**：第一步不是写 Rule，而是先把设计规格文档和 AI 反复磨透；仅有 SPEC 只解决"知道做什么"，解决不了"如何稳定做到位"
3. **七个 Agent 架构**：PM（总路由器）/ 需求分析 / 方案设计 / 闸门总控 / 开发实现 / 代码审查 / 测试验证——被问题一步一步逼出来，非拍脑袋
4. **结构化调度 vs 去中心化协作**：选择结构化调度（固定角色、固定阶段）而非自由协商，因为"真正贵的不是 token，真正贵的是失控"
5. **总验证脚本是核心基础设施**：AI 的完成定义从"我觉得做完了"变成"脚本判定通过才算完成"；建立结果感知能力
6. **Harness 四块拼图**：约束与流程 / 反馈 / 知识库（索引）/ 进化——缺任何一块都会出现典型症状

## 详细摘要

### 第一章：六大零件速览

| 概念        | 它主要回答什么问题       | 本质                   |
| --------- | --------------- | -------------------- |
| Rule      | 什么事绝对不能乱来       | 软约束，原则约束，不能做"流程执行"   |
| Skill     | 这件事具体应该怎么做      | 固定动作标准操作手册，让AI按剧本完成  |
| Sub Agent | 复杂任务由谁分工处理      | 不同阶段专业角色             |
| Workflow  | 这些角色按什么顺序接力     | 前进/暂停/打回/重跑规则，如接力赛   |
| Scripts   | 最后谁来判断到底做没做好    | 硬门禁，客观检查，不靠AI主观汇报    |
| MCP       | AI 怎么安全接上外部工程系统 | 外接能力接口（CI/签名/制品/发布等） |

**关键认知**：Rule 是"告诉 AI 应该怎么做"；Scripts 是"不是告诉 AI，而是直接检查它做没做到"。成熟的 Harness 越来越依赖脚本，而不是越来越依赖提示词。

### 第二章：SPEC 先行原则

> 「我自己在这一步通常会和 AI 来回聊很多轮，甚至聊到有点烦。因为这一轮你偷懒，后面会用十倍时间还回来。」

SPEC 文档要写明：版本要解决什么问题、核心目标与顺手优化的区分、影响哪些模块、哪些行为必须保持兼容、最终什么样才算做完。**必须没有任何不确定词语（如：建议、可以、推荐、可选）**。

### 第三章：Rule 的局限

Rule 多了以后两个典型问题：

- **AI 会忽略 Rule**：复杂任务下"局部遗忘"
- **AI 会绕过 Rule**：找理由解释性执行（"这不是我引入的"/"这次特殊情况"）

Rule 只能做原则约束，不能做流程执行 → 把固定流程下沉成 Skill。

### 第四章：把编译、测试、验证做成 Skill

把 Rule 里的固定流程抽取成 Skill 的好处：

- Rule 变轻（只保留真正红线）
- AI 执行稳定性提高（不是"理解怎么做"，而是"调用现成方法"）
- 维护成本更低（改 Skill 不用遍历所有 Rule）

### 第五章：多 Agent 形态的三条路线对比

| 路线                                 | 最吸引人的地方       | 真正的问题             | 结论                 |
| ------------------------------------ | -------------------- | ---------------------- | -------------------- |
| 继续强化单 Agent                     | 成本最低             | 角色冲突越来越严重     | 早期有效，非最终形态 |
| 去中心化协作（AutoGen GroupChat 类） | 灵活，像AI团队在开会 | 路径不稳定，难长期维护 | 被明确放弃           |
| 结构化调度（固定角色+固定流程）      | 流程清晰，可审计     | 前期设计成本高         | **最终选择**         |

> 「真正贵的不是 token，真正贵的是失控。」

结构化调度内部还有两种做法：固定角色（最终选择）vs 动态招聘 Agent（放弃，因为角色边界漂移）。

### 第六章：七个 Agent 的演化

**第一步**：需求分析 + 方案设计 + 开发实现（最基础骨架）

**第二步**：补闸门总控（可行性分析）——专门站在"开发之前的最后一道门"

**第三步**：代码审查（从实现质量回头看）+ 测试验证（从行为层再看一遍）——两道独立的关

**第四步**：补一个只做路由的 PM——不是"最懂研发的PM"，而是"最守流程边界的PM"

最终七个 Agent：

| Agent          | 职责                                           |
| -------------- | ---------------------------------------------- |
| PM（项目经理） | 路由、交接、回退和进度管理；不越界给专业意见   |
| 需求分析       | 把模糊诉求变成清晰需求                         |
| 方案设计       | 把需求变成技术方案                             |
| 闸门总控       | 开发前最后的可行性和风险把关                   |
| 开发实现       | 落地代码和实现细节                             |
| 代码审查       | 从实现质量、需求一致性、方案一致性做技术收口   |
| 测试验证       | 从功能正确性、稳定性、边界和回归风险做结果收口 |

**实践细节**：不同 Agent 配不同档位的模型——PM 用性价比高的模型，需求分析/代码审查等专业判断重的 Agent 配更强的模型。

### 第七章：六轮补稳过程

| 轮次  | 问题                             | 补法                                                     |
| ----- | -------------------------------- | -------------------------------------------------------- |
| 第1轮 | 下游自己修改上游文档             | 严格规定：下游只能提阻塞项，由PM打回上游                 |
| 第2轮 | PM越界给专业意见                 | PM职责严格收缩为"总路由器"，不负责专业判断               |
| 第3轮 | 各Agent专业性不够                | 强化代码审查和测试的专业性要求（对照需求/方案/验收标准） |
| 第4轮 | Rule约束开始不够用（解释性执行） | 把能判定的约束下沉成脚本                                 |
| 第5轮 | AI偷懒（"不是我引入的"）         | 开发前后各跑一次做基线对比，新增错误必须修               |
| 第6轮 | 流程本身难维护                   | 抽出流程定义文件 + 角色契约 + 流程校验脚本               |

**流程可维护性三层**：

1. **流程定义文件**：有哪些阶段、谁负责、怎么前进/回退
2. **角色契约**：每个Agent必须读哪些上游产物、必须写哪些文档结果
3. **流程校验脚本**：验证流程资产本身是否齐全一致

### 第八章：总验证脚本

总验证脚本的检查项（以 JK Launcher 为例）：

- **A 类（静态规范）**：XAML 中文/Emoji、C#版本限制、MessageBox.Show 禁用、硬编码 UI 文案、日志前缀、SVN 参数规范、文件长度等
- **B 类（交付门槛）**：编译通过、测试全通过、测试数量不能异常减少
- **C 类（工程一致性）**：规则文件多格式同步、.cs 文件进 .csproj

> 「AI 的完成定义不再是'我觉得我做完了'，而变成'脚本判定你通过了，你才算做完'。」

事后验证的核心价值：Harness 第一次具备"结果感知能力"，形成"做事 → 校验结果 → 发现问题 → 回退修复 → 再次验证"的完整回路。

### 第九章：项目级知识库（索引优先）

给 AI 配"跟着仓库走的项目级知识库"，核心是**索引**而非百科全书：

| 模块                        | 由谁维护   | 解决什么问题                                     |
| --------------------------- | ---------- | ------------------------------------------------ |
| **dev-map（开发导航地图）** | 开发 Agent | 某功能落在哪些文件、标准写法是什么、影响哪些链路 |
| **任务看板**                | PM         | 当前哪些任务在做、到什么阶段、文档在哪、历史结论 |

原则：**谁改地貌，谁最有资格改地图**——AI 在流程里自己维护。

### 第十章：Memory 靠边站

> 「事实像档案，例子像故事，规矩像手册——手册应进仓库，别只靠聊天记忆。」

| 类型                       | 更适合落在哪                            |
| -------------------------- | --------------------------------------- |
| 定了的结论、边界、任务索引 | SPEC、任务看板、设计说明、dev-map       |
| 成功/失败案例              | 测试用例、门禁脚本输出约定、评审清单    |
| 反复踩过的坑               | 优先 Scripts（能判定就脚本），其次 Rule |

凡是"团队都要遵守"或"交付要对齐"的东西，不该长期停在 Memory 里。

### 第十二章：Harness 四块拼图

| 拼图            | 管什么                                                        |
| --------------- | ------------------------------------------------------------- |
| **约束与流程**  | AI 按什么顺序、什么边界做事（第3-7章）                        |
| **反馈**        | 做完后系统能不能对结果有说法（第8章）                         |
| **知识库/索引** | 项目级"从哪进门"（第9章）                                     |
| **进化**        | 规范往哪沉淀；人退到哪一层；Harness 怎么继续改版（第10-11章） |

缺任何一块的典型症状：

- 只有流程没有反馈 → "完成幻觉"
- 只有反馈没有流程 → 失败日志无法收敛成可维护改法
- 没有知识库 → 同一类功能反复重写、同一类坑反复踩
- 没有进化 → 前三块冻在某一版，脚本/规则/地图与实际脱节

### 最小起步路径（7步）

1. **先磨 SPEC**（别急着写 Rule）
2. **补最关键的 Rule**（少而精，只盯底线）
3. **高频固定动作 → Skill**（编译/测试/验证）
4. **单 Agent 失稳时再拆多 Agent**
5. **多 Agent 变复杂时补流程定义文件+角色契约**
6. **持续迭代时补 dev-map 和任务看板**
7. **延伸闭环时再考虑 MCP**
