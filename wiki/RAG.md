# RAG

## 定义

RAG（Retrieval-Augmented Generation，检索增强生成）是目前大多数人使用 LLM 处理文档的主流方式：上传文件，模型在查询时检索相关片段，然后生成答案。

## 详细说明

### 工作方式

1. 用户上传一组文件
2. 系统对文件进行分块和索引（通常基于向量嵌入）
3. 查询时，检索与问题最相关的片段
4. LLM 基于检索到的片段生成回答

### 局限性

- **每次从零开始**：没有知识积累，每次提问都要重新发现知识
- **缺乏综合能力**：复杂问题需要综合多篇文档时，模型每次都要重新找到并拼接相关片段
- **无持久化产物**：没有任何东西被沉淀下来

NotebookLM、ChatGPT 文件上传、以及大多数 RAG 系统都属于这种模式。

### 与 LLM Wiki 的对比

[[LLM Wiki]] 提出了不同思路：不在查询时从原始文档检索，而是让 LLM 增量构建持久化 wiki。知识被编译一次并持续更新，而非每次重新推导。

有观点认为，中等规模下（~100篇文章、~40万字），LLM Wiki 模式不依赖传统 RAG，仅靠索引+摘要即可支撑有效检索与推理。

### 上下文工程视角下的 RAG

[[上下文工程]]的研究表明，盲目增加上下文长度反而可能降低模型输出质量（Chroma 的 Context Rot 研究发现安全特性实现下降 47%）。这对 RAG 的启示是：检索策略需要更精细的设计，精准度优先于数量。[[胶水编程]]的实践也验证了这一点——知识库关联采纳率在清理低效条目后从 18% 提升到 35%。
