# 来源：2026 年 AI 编码的"渐进式 Spec"实战指南

## 基本信息

- **作者**：阿里云开发者（阿里云技术社区）
- **发布日期**：2026-04-27
- **来源链接**：https://www.bestblogs.dev/article/fc9e3786
- **核心项目**：code_copilot 框架（实际业务项目中的落地实践）

## 核心要点

1. **"代码廉价，上下文昂贵"**：高质量 Spec 是高密度输入，能将对话轮次从 20 轮降至 3-5 轮，总成本反而更低。
2. **渐进式复杂度**：简单需求直接执行（改字段不写 Spec），复杂需求才引入完整 Propose-Apply-Review-Archive 工作流，压缩偶然复杂度。
3. **Spec 三条铁律**：No Spec No Code（没文档不写代码）、Spec is Truth（文档代码冲突错的是代码）、Reverse Sync（发现 Bug 先修文档再修代码）。
4. **两层 AI 架构**：编排层（强模型如 Claude Opus）负责需求理解和决策；执行层（编码优化模型如 Sonnet/opencode）负责读写代码。
5. **知识底座才是护城河**：AI 编码工具趋同，领域 Know-How、架构决策前因、踩坑经验才是决定 AI 输出质量上限的不可复制资产。

## 详细摘要

### 一、背景：三个基础认知

- **理解模型**：顶级模型能完成中等复杂编码任务，但无持久记忆。梯队差异体感明显——T0 三轮完成，T2 可能 15 轮不对。**模型是地基，方法论是上层建筑**。
- **理解 Agent**：`Agent = while 循环 + Tool Use + 工具执行器`。工具边界即能力边界，安全靠框架约束不靠 AI 自觉。
- **回归本质**：引用《人月神话》——软件复杂度 = 本质复杂度（不可消除）+ 偶然复杂度（工具引入的额外负担，应被消除）。方法论好坏的标准：高效应对本质复杂度，同时自身引入的偶然复杂度最低。

### 二、渐进式编码框架（code_copilot）

**Spec Coding** 是框架核心：先用结构化文档把"要做什么、怎么做、有什么约束"说清楚，再让 AI 围绕文档编码。

**为什么需要 Spec**：直接 Vibe Coding 面临四大工程问题：

1. AI 总是漏需求（自然语言歧义）
2. 代码没有架构感（缺乏结构化信息）
3. 上下文越长越降智（错误信息污染）
4. 写码快不代表深思熟虑（速度掩盖思考缺失）

**渐进式复杂度**（框架最大亮点）：

| 需求复杂度                     | 流程深度                           |
| ------------------------------ | ---------------------------------- |
| 简单（≤5 人日，改字段/修 bug） | 直接执行，Rules 生效，无 Spec 成本 |
| 中等                           | 轻量 Spec + Tasks                  |
| 复杂                           | 完整 Propose-Apply-Review-Archive  |

**目录结构**：

```
code_copilot/
├── rules/          # 始终生效（项目规范、编码风格、安全红线）
├── knowledge/      # 按需加载（领域知识索引 + 详细文档）
├── agents/         # Agent 配置与提示词
├── changes/        # 变更管理（每需求一目录，含 spec/tasks/log）
└── archives/       # 已归档变更
```

**知识飞轮**：需求实践 → 踩坑 → 沉淀 knowledge/更新 prompt → AI 更准 → 更好实践（prompt 和模板自身也在飞轮中）

### 三、工作流详解

**Propose（提案）**：Research 代码现状（必须有出处）→ 逐个提问（一次一问，给推荐选项）→ 分段生成 spec.md + tasks.md + log.md → 确认门控 HARD-GATE（显式确认后才能编码）

**Apply（执行）**：默认逐步执行（完成一 task 等确认）→ 零偏差原则（Plan 是合同，AI 是打印机）→ Verification 铁律（每个 task 必须展示可验证证据，禁止"应该没问题"）

**Fix（修正）**：Apply 和 Review 之间的增量修正，每次必须同步更新 spec/tasks/log

**Review（审查）**：Sub Agent 两阶段独立审查（与实现者上下文隔离）：

1. Spec Compliance（逐条比对 spec 与实际代码，"不信报告只信代码"）
2. Code Quality（基于 rules/ 检查，Critical/Important/Minor 分级）

**Archive（归档）**：逐条展示知识发现，确认沉淀到 knowledge/，变更目录移入 archives/

### 四、两层 AI 架构

```
人（开发者）→ 编排层 AI（强模型：需求理解/Spec 生成/审查决策）
                        ↓
             执行层 AI（编码模型：读写代码/执行命令/快速迭代）
```

分层原因：关注点分离，而非只是成本考虑。混合会导致"要么太贵要么质量不够"。

透明度是工具选型底线：模型型号+版本可见、完整 context 可查、原始输出不被篡改、token 用量透明。

### 五、深层思考

**人的角色转变**：传统编程（设计者+实施者+验收者）→ AI 编程（设计者+验收者，AI=实施者）。四大角色：AI 侦察兵 → 参谋 → 施工队 → 质检员。

**自由度给反了的普遍问题**：应该在方案设计阶段给最高自由度，执行阶段给零自由度；大部分人反过来做（方案没想清楚就开写，执行时又让 AI 自由发挥）。

**知识底座是真正护城河**：Spec 框架最难覆盖的恰恰是最重要的知识——领域 Know-How、架构决策前因、团队隐性经验。这才是不可复制的护城河，优先级高于优化 Prompt/Rules/Agent 工作流。

## 引用原文关键段落

> 代码廉价，上下文昂贵（Code is Cheap, Context is Expensive）——把需求、约束、代码现状写进 Spec 作为高质量输入 → 输入增加但便宜 → AI 不用反复试错 → 输出大幅减少 → 对话轮次从 20 轮降到 3-5 轮 → 总成本反而更低，效果反而更好。

> AI 编码工具会越来越同质化（Cursor、Claude Code、各种 IDE 插件的能力趋同），团队之间的差距不在于用什么工具，而在于积累了多少高质量的、结构化的领域知识。这才是真正不可复制的护城河。

> 大部分人的问题是反过来了——该讨论的时候急着让 AI 干活（方案没想清楚就开写），该干活的时候又让 AI 自由发挥（执行阶段不约束，改着改着就跑偏了）。
