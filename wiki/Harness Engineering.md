# Harness Engineering

## 定义

Harness Engineering（马具工程/驾驭工程）是由 Terraform 作者 Mitchell Hashimoto 于 2026 年提出的 AI 编程方法论：每次发现 AI 犯错，就花时间工程化一个机制，让它以后再也不犯这个错。核心是构建环境让 AI 可靠地工作，包括验证闭环、架构约束、测试护栏。

## 详细说明

### AI 编程工具演进路线

Harness Engineering 是 AI 编程工具演进的第三阶段：

| 时期      | 阶段                                | 核心问题                   | 本质                         |
| --------- | ----------------------------------- | -------------------------- | ---------------------------- |
| 2023-2024 | Prompt Engineering                  | 怎么跟 AI 说话             | 优化一次性自然语言输入       |
| 2025      | [[上下文工程\|Context Engineering]] | 给 AI 看什么信息           | 设计整个信息环境             |
| 2026      | **Harness Engineering**             | 构建什么环境让 AI 可靠工作 | 验证闭环、架构约束、测试护栏 |

从 Prompt 到 Context 到 Harness，方向是**从自然语言走向形式化约束**——这恰好验证了 [[Edsger Dijkstra]] 48 年前的洞察。

### 形式化约束的具体形态

在 AI 编程时代，形式化约束体现为：

- **CLAUDE.md / AGENTS.md**：写给 AI 的形式化规范
- **TDD 测试套件**：用代码表达的验收标准
- **CI/CD 管道**：自动化的质量闸门
- **类型系统、接口定义**：结构化的契约

过去 CI/CD 集成、TDD 被认为是昂贵的工程实践，很多团队嫌麻烦不做。但在 AI 编程时代，它们已从奢侈品变成了必需品。

### 与胶水编程的互补关系

[[胶水编程]]的四层物料体系（开发规范、代码模式、领域知识、任务规格）可以视为 Harness Engineering 在中后台开发领域的具体落地：开发规范是形式化的编码约束，代码模式是形式化的结构模板，领域知识是形式化的经验沉淀。

### AI 作为形式化的桥梁

Harness Engineering 并非否定自然语言的价值，而是确立了三层结构：

- **自然语言**是输入层——降低表达意图的门槛
- **形式化**是验证层——保证意图被正确执行
- **AI**是翻译器——让形式化的生产成本大幅降低

> 写代码变便宜了，但思考没有变便宜。

## 工程实践：阿里云开发者 code_copilot 个人框架

阿里云开发者将 [[Spec Coding]] 与 Harness Engineering 理念融合，设计了 code_copilot 个人框架。它展示了 Harness 四块拼图在个人/小团队场景中的最小实现路径。

| Harness 四块拼图 | code_copilot 对应实现                                                      |
| ---------------- | -------------------------------------------------------------------------- |
| 约束与流程       | `rules/`（始终生效）+ Propose-Apply 工作流 + HARD-GATE 门控                |
| 反馈             | Verification 鐵律（每个 task 展示可验证证据）+ Sub Agent Review 两阶段审查 |
| 知识库/索引      | `knowledge/`（触发关键词索引）+ `changes/log.md`（实时踩坑采集）           |
| 进化             | 知识飞轮（需求实践→踩坑→沉淀 knowledge/→AI 更准）+ Archive 归档机制        |

## 工程实践：白家杰 JK Launcher 落地全记录

[[\u767d\u5bb6\u6770]] 以 JK Launcher（Unity 桌面启动器）为真实案例，记录了从零搞起 Harness Engineering 的完整演化路径。

### Harness 六大零件

| 零件          | 解决什么问题                | 本质                                            |
| ------------- | --------------------------- | ----------------------------------------------- |
| **Rule**      | 什么事绝对不能乱来          | 软约束，只能做“原则约束”，不能做“流程执行”      |
| **Skill**     | 这件事具体应该怎么做        | 固定动作标准操作手册，AI 按剧本而非纯ug发挥完成 |
| **Sub Agent** | 复杂任务由谁分工处理        | 不同阶段的专业角色                              |
| **Workflow**  | 这些角色按什么顺序接力      | 前进/暂停/打回/重跑规则，如接力赛               |
| **Scripts**   | 最后谁来判断到底做没做好    | 硬门禁，客观检查，不靠 AI 主观汇报              |
| **MCP**       | AI 怎么安全接上外部工程系统 | 外接能力接口（CI/签名/制品/发布等）             |

**关键认知**：Rule 是“告诉 AI 应该怎么做”；Scripts 是“直接检查它做没做到”。成熟的 Harness 越来越依赖脚本，而不是越来越依赖提示词。

### SPEC 先行原则

第一步不是写 Rule，而是先把设计规格文档（SPEC）与 AI 反复磨透。SPEC 要写明版本目标、边界条件、影响模块、兼容要求、验收标准。**必须没有任何不确定词语（如：建议、可以、推荐、可选）**。

> 「这一轮你偷懒，后面会用十倍时间还回来。」——白家杰

### 三种多 Agent 形态对比

| 路线                                 | 最吸引人的地方         | 真正的问题             | 结论                 |
| ------------------------------------ | ---------------------- | ---------------------- | -------------------- |
| 继续强化单 Agent                     | 成本最低               | 角色冲突越来越严重     | 早期有效，非最终形态 |
| 去中心化协作（AutoGen GroupChat 类） | 灵活，像 AI 团队在开会 | 路径不稳定，难长期维护 | **被明确放弃**       |
| 结构化调度（固定角色+固定流程）      | 流程清晰，可审计       | 前期设计成本更高       | **最终选择**         |

> 「真正贵的不是 token，真正贵的是失控。」

### 七个 Agent 的演化步骤

被真实问题一步步逆出来，非拍脑袋：

| Agent              | 职责                                           | 必要性                       |
| ------------------ | ---------------------------------------------- | ---------------------------- |
| **PM（项目经理）** | 路由/交接/回退/进度管理                        | “总路由器”，不越界给专业意见 |
| **需求分析**       | 把模糊诉求变成清晰需求                         | 防止需求直接流进代码         |
| **方案设计**       | 把需求变成技术方案                             | 需求和实现必须分开           |
| **闸门总控**       | 开发前最后的可行性和风险把关                   | 把问题提前暴露，穿展开发最贵 |
| **开发实现**       | 落地代码和实现细节                             | 只按前面的约束去写           |
| **代码审查**       | 从实现质量、需求一致性做技术收口               | 不能只看“本层有没有问题”     |
| **测试验证**       | 从功能正确性、稳定性、边界和回归风险做结果收口 | 与代码审查是两道独立的关     |

**模型分层配置**：PM 等路由角色用性价比高的模型；需求分析、代码审查等专业判断重的 Agent 配更强模型。

### 六轮补稳过程

| 轮次  | 问题                      | 补法                                    |
| ----- | ------------------------- | --------------------------------------- |
| 第1轮 | 下游自己修改上游文档      | 下游只能提阻塞项，由 PM 打回上游        |
| 第2轮 | PM 越界给专业意见         | PM 职责收缩为“总路由器”，不负责专业判断 |
| 第3轮 | 各 Agent 专业性不够       | 强化代码审查和测试的专业性要求          |
| 第4轮 | Rule 被解释性执行         | 把能判定的约束下沉成脚本                |
| 第5轮 | AI 偶懒（“不是我引入的”） | 开发前后各跑一次做基线对比              |
| 第6轮 | 流程本身难维护            | 流程定义文件 + 角色契约 + 流程校验脚本  |

### 总验证脚本：Harness 最关键的基础设施

总验证脚本是“研发后验证总入口”，它发知答“这次开发到底合不合格”：

- **A 类**：静态规范问题（禁用 API、语法版本限制、更编码等）
- **B 类**：基础交付门槛（编译通过、测试全通过、测试数量不能异常减少）
- **C 类**：工程一致性（规则文件同步、文件进工程配置）

> 「AI 的完成定义不再是'我觉得我做完了'，而变成'脚本判定你通过了，你才算做完'。」

事后验证的核心价值：Harness 第一次具备“结果感知能力”，形成“做事 → 校验结果 → 发现问题 → 回退修复 → 再次验证”的完整回路。

### 项目级知识库（索引优先）

给 AI 配“跟着仓库走的项目级知识库”，核心是**索引**而非百科全书：

| 模块                        | 由谁维护   | 解决什么问题                                                 |
| --------------------------- | ---------- | ------------------------------------------------------------ |
| **dev-map（开发导航地图）** | 开发 Agent | 某功能落在哪些文件、标准写法是什么、改一个模块会牺动哪些链路 |
| **任务看板**                | PM         | 当前哪些任务在做、到什么阶段、文档在哪、历史结论是什么       |

**原则**：谁改地貌，谁最有资格改地图——AI 在流程里自己维护。

### Harness 四块拼图

| 拼图            | 管什么                       | 缺一块的典型症状                           |
| --------------- | ---------------------------- | ------------------------------------------ |
| **约束与流程**  | AI 按什么顺序、什么边界做事  | 模型再强也只是在对话里即兴发挥             |
| **反馈**        | 做完后系统能不能对结果有说法 | “完成幻觉”：看起来很忙但不知道做对没有     |
| **知识库/索引** | 项目级“从哪进门”             | 同一类功能反复被重写、同一类坑反复踩       |
| **进化**        | 规范往哪沉淠；人退到哪一层   | 前三块冻在某一版，脚本/规则/地图与实际脱节 |

### 最小起步路径（7步）

1. **先磨 SPEC**（别急着写 Rule）
2. **补最关键的 Rule**（少而精，只盯底线）
3. **高频固定动作 → Skill**（编译/测试/验证）
4. **单 Agent 失稳时再拆多 Agent**
5. **多 Agent 变复杂时补流程定义文件+角色契约**
6. **持续迭代时补 dev-map 和任务看板**
7. **延伸闭环时再考虑 MCP**

### Memory 靠边站

> 「事实像档案，例子像故事，规矩像手册——手册应进仓库，别只靠聊天记忆。」

凡是“团队都要遵守”或“交付要对齐”的东西，不该长期停在 Memory 里。组织百托的坏、绝不能碰的红线，应优先进 Scripts（能判定就脚本化），其次进 Rule。

## 工程实践：腾讯 CDN LEGO 五层架构

腾讯 CDN LEGO 团队（百万行 C++ 系统）在 2026 年将 Harness Engineering 落地为**五层架构**，围绕三大核心要素构建：

### 核心要素：上下文、约束、反馈

**1. 上下文建设**（消除 AI 的"记忆偏差"）

四层递进体系：

- Agent.md（项目宪法）：项目结构即上下文，架构模式即约束
- 安全纪律：用"反例免疫"替代正面说教，每条规则都有错误写法和正确写法
- 可复用模式库：CR 检查清单来自真实问题且经 A/B 验证
- 专业 Skill：覆盖友商实现、协议 RFC、开源代码等领域知识

特别实践：将 38,068 行 RFC 原文固化在本地——AI 通过直接读取而非"回忆"来引用协议标准，避免引用过时版本（如 RFC 7230/7231 已被 9110/9112 取代）。

**2. 约束**（用结构化约束替代语言化期望）

明确的约束比模糊的期望更有效：

| 方式    | 示例                              | 效果                    |
| ------- | --------------------------------- | ----------------------- |
| ❌ 期望 | "写高质量的代码"                  | AI 理解模糊，输出不稳定 |
| ✅ 约束 | "禁止裸 new，必须 unique_ptr"     | AI 100% 遵循            |
| ❌ 期望 | "记得写测试"                      | 8 个 Agent 忘了测试     |
| ✅ 约束 | TaskList 中测试 Task 阻塞后续流程 | 不可能跳过              |

**3. 反馈**（反馈速度决定进化速度）

进化闭环：真实踩坑 → 安全规则 → CR 检查清单 → A/B 验证 → 保留或标注

LEGO 的三条并行反馈通道：自动采集（Hook）、踩坑日志（Pitfall Journal）、CLAUDE.md 内联反馈。

### AI Coding 常见问题分类

LEGO 团队基于 57 个真实案例提炼 13 类问题，最危险的是：

> **不会说"我不知道"**——AI 会用自信的语气输出错误结论，反而降低人的审查意愿

其他高危问题：幻觉（编造 API/RFC 章节号/数据）、改不全（局部修改遗漏全局影响）、模式匹配代替验证、缺乏环境意识。

### [[对抗式CR]]：多模型并行审查

三模型（Claude + Codex + Gemini）**并行独立审查** → 交叉验证 → 辩论式讨论（同意/反对/维持）→ 全员无新发现时自动收敛。将代码审查等待时间从 1-3 天压缩至 30 分钟。

### 实践收益（LEGO 阶段性数据）

综合效率提升 20%（含人工 Review + 学习曲线）。局部提升：竞品调研 ~3x、协议安全测试 ~4x。

剩余挑战：误报率 36%、文档爆炸、AI 自信会传染、团队能力退化风险。

## 工程实践：腾讯审核团队全链路自动化

另一腾讯团队将 Harness Engineering 理念嵌入全链路 AI 自动化框架，提出公式：

> **AI 全自动化交付 = LLM + Harness Engineering（工作流程 + 知识库 + Skills 管理）+ Business Engineering（业务专属 Skills + 知识库）**

通过三阶段演进（L1 纯人工 → L2 人机协同 → L3 全自动），最终目标是需求到部署的全自动化交付。

详见 → [[来源_腾讯审核团队全链路AI自动化]] 和 [[AI全自动化交付]]

## 工程师角色演变

在 Harness Engineering 时代，工程师角色系统性重构：

- 初级开发 → **AI 操作员**（掌握 Skill 与 Prompt）
- 高级开发 → **Harness 工程师**（设计 AI 的约束、上下文与规则）
- 架构师 → **人机协作架构**（判断什么交给 AI、什么必须人来把控）
- 测试/安全 → **AI 质量/安全专家**

不变的核心能力：**抽象思维**——知道什么该让 AI 做、如何验证 AI 做得对。
