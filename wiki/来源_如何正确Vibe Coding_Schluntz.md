# 来源：如何正确 Vibe Coding？——Schluntz 演讲

## 基本信息

- **演讲者**：[[Erik Schluntz]]，Anthropic 研究员，《构建高效智能体》合著者
- **整理发布**：微信公众号"Vibe Coding的"，2026-04-20
- **来源链接**：https://mp.weixin.qq.com/s?__biz=MzA3MzI4MjgzMw==&mid=2651028728
- **背景**：Schluntz 因摔断手臂、打了两个月石膏，被迫将全部工作交给 Claude，由此积累了生产环境 Vibe Coding 的深度实战经验

## 核心要点

1. **叶子节点策略**：将 Vibe Coding 限制在代码库的"叶子节点"——不被其他任何模块依赖的末端功能；主干架构仍需人工深入理解和严密保护
2. **可验证抽象层**：开发者要找到"无需阅读底层代码即可验证正确性"的抽象层，如验收测试、E2E 测试、关键数据切片，类比 CTO/PM/CEO 各自的验证方式
3. **AI 的 PM 工作流**：在让 Claude 写代码前，先花 15-20 分钟探索代码库、制定计划、汇总上下文，任务成功率呈指数级提升
4. **极简 TDD 原则**：Vibe Coding 中使用极简 E2E 测试（3 个测试：happy path + 两个错误场景），避免 Claude 陷入过度测试；唯一需要审查的代码就是测试
5. **上下文压缩技巧**：在自然停顿点 Compact 上下文；先让 Claude 将计划写成文档，再立刻压缩，将 10 万 token 压缩为几千 token

## 详细摘要

### Vibe Coding 的精准定义

Schluntz 引用 Karpathy 的定义强调，Vibe Coding 不是"重度使用 AI 工具并逐行审查"，而是：

> 「完全沉浸于氛围，拥抱技术发展的指数级增长，并且彻底忘记代码的存在。」

真正的 Vibe Coding 的核心原则是：**忘记代码的存在，但必须始终关注产品的存在。**

### 叶子节点（Leaf Nodes）策略

为应对技术债难以量化的问题，Schluntz 提出聚焦代码库中的叶子节点：

- **叶子节点**：不被其他任何模块依赖的末端功能或附加组件
- 在这些区域产生一定技术债是可接受的（极少变动，不阻碍其他模块）
- 系统主干与底层架构仍需人工深入理解，保护可扩展性
- 随模型能力提升，可信任 AI 接管的层级正在动态下移

### 可验证抽象层

参照企业管理的方式理解：

- CTO 用**验收测试**管理技术专家
- PM 通过**体验产品**验证功能
- CEO 借**数据切片**抽查财务模型

工程师需要建立类似的可验证抽象层，而 TDD 测试套件正是这种抽象层的工程实现。

### 22000 行代码生产合并案例

Anthropic 内部真实案例：强化学习代码库中合并 22000 行代码（绝大部分由 Claude 编写），原本需要两周的工程压缩到 **1 天**内完成。四大核心策略：

1. **PM 视角深度引导**：数天时间进行前期人工规划与需求梳理
2. **严格限制在叶子节点**：将代码变更限定在允许存在技术债的区域
3. **核心区域人工介入**：底层扩展性相关逻辑严格人工审查
4. **建立可验证检查点**：长时间压力测试 + 易于人类验证的输入/输出标准

### AI 的 PM 标准前置工作流

```
1. 让 Claude 探索代码库、找到相关文件
2. 共同制定清晰的执行计划
3. 将上下文和规范汇入一个单独的 prompt
4. 执行
```

全程约 15-20 分钟，但任务成功率指数级提升。

### TDD 实践技巧

- 强制规范：「只写 3 个 E2E 测试：happy path + 错误场景 1 + 错误场景 2」
- 引导写极简端到端测试，而非过度依赖具体实现的单元测试
- Vibe Coding 时唯一需要审查的代码就是测试代码

### 上下文压缩（Compact）技巧

- 在感觉到"人类程序员会停下来吃午饭"的自然停顿点进行 Compact
- 起手式：先让 Claude 找出相关文件并制定计划，将计划全部写入文档，立刻 Compact
- 效果：将 10 万个制定计划耗费的 token，压缩为几千个干净的 token
