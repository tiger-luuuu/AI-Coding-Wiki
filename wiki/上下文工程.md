# 上下文工程

## 定义

上下文工程（Context Engineering）是指精心设计和管理提供给 LLM 的上下文信息的工程实践——决定什么信息在什么时机、以什么方式进入模型的上下文窗口，以最大化模型输出质量。

## 详细说明

### 核心原则

**精准度优先于数量**。Anthropic 上下文工程博客的核心观点：

> 一个精准的 300 token 上下文，往往胜过一个混杂的 113,000 token 上下文。

Chroma 的 Context Rot 研究也验证了这一点：在 18 个 SOTA 模型的测试中，上下文越长，安全特性实现反而下降了 47%。

### 静态注入 vs 动态检索

上下文工程的核心设计决策之一是信息的加载时机：

- **静态注入**：关键规则始终在场，不依赖 AI 的主动调用决策。Vercel 评测显示，56% 的场景中 AI 不会主动调用文档工具；将同样内容静态加载后，通过率从 53% 提升到 100%
- **动态检索**：补充性知识按需拉取，避免上下文过载导致的性能退化

[[胶水编程]]的四层物料体系是这一原则的具体工程实践：开发规范和代码模式静态注入（始终在场），领域知识动态检索（按需拉取），任务规格按需生成。

### 与知识管理的关系

上下文工程的理念与 [[LLM Wiki]] 的设计哲学一脉相承：

- LLM Wiki 通过预先编译知识（摘要、交叉引用、索引），减少每次查询时 LLM 需要处理的原始信息量
- [[胶水编程]]通过分层物料体系，在正确的时机注入正确粒度的上下文
- 两者都在解决同一个问题：**如何让 LLM 在有限的上下文窗口中获得最高密度的有效信号**

### 在 AI 编程工具演进中的位置

上下文工程是 AI 编程工具演进三阶段的中间阶段：

- 2023-2024：Prompt Engineering — 优化自然语言输入
- **2025：Context Engineering** — 设计整个信息环境
- 2026：[[Harness Engineering]] — 构建形式化约束环境

方向是从自然语言走向形式化约束。[[Edsger Dijkstra]] 在 1978 年就预见了这一趋势——形式化符号是排除胡说八道的工具。

### 与 RAG 的对比

传统 [[RAG]] 的做法是把大量检索结果一股脑塞入上下文，上下文工程的视角则强调：不是检索越多越好，而是要精准匹配当前任务所需的信息层级和粒度。

### 前置规划工作流（Pre-planning Workflow）

[[Erik Schluntz]] 提出的 AI 的 PM 标准前置工作流，是上下文工程在 Vibe Coding 场景中的具体实现：

```
1. 让 Claude 探索代码库、找到相关文件
2. 共同制定清晰的执行计划
3. 将上下文和规范汇入一个单独的 prompt
4. 执行
```

前置投入 15-20 分钟规划，任务成功率呢指数级提升。关键原则：当不对模型施加过度约束时，模型表现最好——把它当成初级工程师沟通就好。

### 上下文压缩（Context Compact）技巧

Vibe Coding 长程序运作中，上下文应在何时压缩是一个实践级问题。Schluntz 的方法：

- **压缩时机**：在感觉到“人类程序员会停下来吃午饭”的自然停顿点进行 Compact
- **起手式**：先让 Claude 找出所有相关文件并制定计划，将这些全写进一个文档，立即进行 Compact
- **效果**：将制定计划耗费的 10 万个 token 压缩为几千个干净的 token

这与上文提到的“知识预编译”思想一脏相承：不让模型在运行时重新推导，而是提前将计划结果压缩为高密度上下文。
