# Spec Coding

## 定义

Spec Coding 是一种在让 AI 写代码之前，先用结构化文档（Spec）将"要做什么、怎么做、有什么约束"明确说清楚，然后让 AI 围绕这份文档进行编码的方法论。其核心理念是：**代码廉价，上下文昂贵（Code is Cheap, Context is Expensive）**。

Spec Coding 是 [[Vibe Coding]] 的对立面与进化——Vibe Coding 以自然语言驱动 AI 即兴生成，Spec Coding 以结构化文档约束 AI 按图施工。在 AI 编码可控性递进中，Spec Coding 处于 Vibe Coding（随机）与 [[胶水编程|Glue Coding]]（从已有代码组装）之间的中间阶段。

## 详细说明

### 为什么需要 Spec Coding

直接与 AI 对话写代码（Vibe Coding）面临四大工程问题：

1. **AI 总是漏需求**：自然语言让人轻松说出自己都没意识到的模糊和矛盾，AI 理解的和实际意图之间存在系统性偏差
2. **代码没有架构感**：自然语言描述中没有分层、依赖关系、接口设计的形式化信息，生成的代码能跑但结构松散
3. **上下文越长越降智**：对话一长，AI 被之前的错误信息污染，在错误方向上越走越远
4. **写码快不代表深思熟虑**：速度掩盖了思考的缺失，快速产出掩盖了设计上的不足

高质量的 Spec 作为高密度输入，可以将对话轮次从 20 轮压缩到 3-5 轮，总成本反而更低。

### 三条铁律

1. **No Spec, No Code**：没有文档，不准写代码
2. **Spec is Truth**：文档和代码冲突时，错的一定是代码（代码是文档的实现，不是反过来）
3. **Reverse Sync**：发现 Bug 或实现偏差，先修文档，再修代码

### 渐进式复杂度（核心设计）

Spec Coding 的最大亮点是**渐进式复杂度**——不同复杂度的需求，暴露不同深度的流程，本质上是在压缩偶然复杂度：

| 需求复杂度      | 典型场景           | 流程深度                               |
| --------------- | ------------------ | -------------------------------------- |
| 简单（≤5 人日） | 改字段、修 Bug     | 直接执行，Rules 始终生效，无 Spec 开销 |
| 中等            | 新增独立功能       | 轻量 Spec + Tasks                      |
| 复杂            | 跨模块迁移、新系统 | 完整 Propose-Apply-Review-Archive      |

> 70% 的需求是 ≤5 人日的小需求，改个字段也要先写 Spec 再拆 Tasks，这就是偶然复杂度在吃掉效率。

### Propose-Apply-Review-Archive 工作流

**Propose（提案）— 人主导，AI 辅助**

- Research：AI 分析代码现状，每个结论必须有代码出处（文件路径 + 类名/方法名）
- 逐个提问：一次只问一个问题，优先给 2-3 个选项 + 推荐，减少用户思考负担，同时做 YAGNI 裁剪
- 分段生成文档，每段确认后继续：越早发现方向偏差，修正成本越低
- 确认门控 HARD-GATE：必须等用户显式确认后才能进入编码

**Apply（执行）— AI 主导，人审查**

- 默认逐步执行：完成一个 task → 报告 → 等用户确认（或授权批量执行）
- 零偏差原则：Plan 是合同，AI 是打印机——遇到逻辑冲突立即停止，先 Reverse Sync
- Verification 铁律：每个 task 完成后必须展示可验证证据（编译输出/测试输出/调用结果），禁止"应该没问题"等无证据声明

**Fix（修正）— 增量迭代**

- 填补 Apply 和 Review 之间的修正环节
- 每次 Fix 必须同步更新 spec、tasks、log

**Review（审查）— 两阶段 Sub Agent**

通过独立 Sub Agent 执行（与实现者上下文隔离，避免主观性偏差）：

1. **Spec Compliance**：逐条比对 spec 功能点与实际代码，"不信报告只信代码"
2. **Code Quality**：基于 rules/ 检查编码规范、安全红线、异常处理，按 Critical/Important/Minor 分级

阶段一 PASS 后才启动阶段二，任一 FAIL 则回到 Apply/Fix 修正。

**Archive（归档）— 知识沉淀**

逐条展示 log.md 中的知识发现，确认沉淀到 knowledge/，变更目录移入 archives/。

### code_copilot 框架目录结构

将 Spec Coding 落地的完整工程目录：

```
code_copilot/
├── rules/                      # Project Rules（始终生效）
│   ├── project-context.md      # 工程结构、分层、核心依赖
│   ├── coding-style.md         # 编码规范
│   ├── security.md             # 安全红线
│   └── domain-rules.md         # 业务领域约束
│
├── knowledge/                  # 领域知识（按需加载）
│   ├── index.md                # 知识索引（触发关键词 + 描述）
│   └── *.md                    # 详细知识文档
│
├── agents/                     # Agent 配置与提示词（可迭代）
│   ├── copilot-prompt.md       # 主 Agent 提示词
│   ├── spec-reviewer.md        # Spec 合规审查 Agent
│   └── code-quality-reviewer.md # 代码质量审查 Agent
│
├── changes/                    # 变更管理（每需求一目录）
│   ├── templates/              # 模板（spec/tasks/log）
│   └── <change-name>/          # spec.md + tasks.md + log.md
│
└── archives/                   # 已完成变更的归档
```

**关键设计**：rules/ 始终生效（静态注入），knowledge/ 按需加载（动态检索），两者分工明确，对应[[上下文工程]]的核心原则。

### 两层 AI 架构

Spec Coding 实践中演化出的编排+执行双层架构：

```
人（开发者）
  ├── 编排层 AI（强模型，如 Claude Opus）
  │   职责：理解模糊需求、生成 Spec、跨仓库分析、审查决策
  │       ↓
  └── 执行层 AI（编码优化模型，如 Sonnet/opencode）
      职责：读写代码、执行 shell 命令、快速迭代修改
```

分层的核心价值不仅是成本，更是**关注点分离**：混合两种角色会导致"要么太贵要么质量不够"。

工具选型的透明度底线：模型型号+版本可见、完整 context 可查、原始输出不被篡改、token 用量透明。

### 知识底座：真正的护城河

Spec 框架能覆盖的知识和真正重要的知识存在明显落差：

| 知识类型     | Spec 能覆盖 | 实际重要性 |
| ------------ | ----------- | ---------- |
| 编码规范     | ★★★★        | ★★★        |
| 存量代码     | ★★★         | ★★★★       |
| 领域知识     | ★           | ★★★★★      |
| 架构决策     | ★★          | ★★★★★      |
| 团队隐性经验 | ☆           | ★★★★       |

> 一个没有 knowledge/ 的 Spec 框架，就像让一个刚入职的应届生对着编码规范写代码——规范他都能遵守，但业务逻辑全靠猜。

AI 编码工具越来越同质化，团队之间的差距在于**积累了多少高质量的、结构化的领域知识**，而非在于用什么工具。这是 Spec Coding 和 [[LLM Wiki]] 理念的汇聚点：**知识编译一次，持续复用**。

### Reverse Sync：最容易被忽略的核心

发现代码偏差或 Bug 时，大部分人的本能是直接改代码。Spec Coding 的铁律是反过来：

1. 确认事实（AI 先 Research 代码确认偏差来源）
2. 先修文档（更新 spec.md 体现正确意图）
3. 再修代码（按更新后的 spec 修正实现）

不执行 Reverse Sync，文档和代码会很快脱节，整个 Spec 框架就会失效。

### 人的角色转变与自由度管理

```
传统编程：人 = 设计者 + 实施者 + 验收者
AI 编程：人 = 设计者 + 验收者，AI = 实施者
```

AI 在不同阶段的角色：侦察兵（扫描代码收集事实）→ 参谋（提出方案分析利弊）→ 施工队（按图施工）→ 质检员（对照标准检查）

**自由度曲线**（普遍的反向错误）：

| 阶段     | 正确自由度                       | 常见错误                     |
| -------- | -------------------------------- | ---------------------------- |
| 调研     | 中（自由探索但必须给证据）       | 太早限制 AI 探索             |
| 方案设计 | 高（唯一鼓励 AI 充分想象的阶段） | ✓                            |
| 规划     | 低（精确到文件路径和函数签名）   | 该严格时不严格               |
| 执行     | 零（严格按计划施工，有问题立停） | **给了太大自由度，导致跑偏** |
| 验收     | 中（自由检查但结论要有依据）     | ✓                            |

## 与 Harness Engineering 的关系

Spec Coding 与 [[Harness Engineering]] 是同一方向上的不同粒度：

- Harness Engineering 是宏观工程哲学：构建形式化约束环境让 AI 可靠工作
- Spec Coding 是微观方法论：以结构化文档为中心的单需求工作流

code_copilot 框架中的 rules/（约束层）、knowledge/（知识库）、Sub Agent Review（反馈层）、知识飞轮（进化层）与 Harness 四块拼图（约束与流程、反馈、知识库/索引、进化）一一对应。
