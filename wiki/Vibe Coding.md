# Vibe Coding

## 定义

Vibe Coding 是一种用自然语言直接描述需求、让 AI 生成代码的编程方式。开发者不需要精确定义架构和实现细节，而是凭"感觉"（vibe）与 AI 对话，快速产出可运行的代码。

该术语由 [[Andrej Karpathy]] 推广，他曾表示自己从 2025 年 12 月起几乎没再亲手敲过代码。

## 详细说明

### 甜蜜期与局限

Vibe Coding 的甜蜜期是真实的：用自然语言描述需求，AI 几分钟就能生成一个能跑的 demo，适合探索原型和快速验证。

但在生产环境中，其结构性问题逐步浮现：

- **AI 总是漏需求**：你以为说清楚了，AI 也表现得好像理解了，但交付的东西总是差几个关键点。本质上是自然语言让人轻松说出自己都没意识到的模糊和矛盾
- **代码没有架构感**：AI 生成的代码能跑但结构松散，"帮我写一个用户管理模块"这句话里没有任何关于分层、依赖关系、接口设计的形式化信息
- **上下文越长越降智**：对话一长，AI 被之前的错误信息污染，在错误方向上越走越远
- **写码快不代表深思熟虑**：速度掩盖了思考的缺失

这些问题在 1978 年就被 [[Edsger Dijkstra]] 预见。他在 EWD667 手稿中指出：自然语言的"自然性"不过是让人能轻松说出荒谬性并不显而易见的话。

### 在 AI 编码可控性递进中的位置

[[胶水编程]]将 Vibe Coding 定位为 AI 编码可控性的第一阶段：

| 阶段            | 解决什么                | 局限                             |
| --------------- | ----------------------- | -------------------------------- |
| **Vibe Coding** | AI 能不能写代码         | 产出不可控，风格随机，质量看运气 |
| [[Spec Coding]] | AI 写的代码对不对       | 管意图和边界，管不到具体实现     |
| Glue Coding     | AI 写的代码像不像我们的 | 需要完整物料体系支撑             |

### 从 Vibe Coding 到形式化的回归

实践者发现，Vibe Coding 之后的自然演化不是更多的自然语言，而是回归形式化约束。典型的 Planned 模式开发流程（需求 → 对齐 → 细化 → 预研 → 测试设计 → 系统设计 → 任务分拆 → 执行），每一步都在把自然语言的模糊描述逐步转化为更精确的约束。

AI 编程工具的演进也印证了这一趋势：

- 2023-2024：Prompt Engineering — 优化自然语言输入
- 2025：[[上下文工程|Context Engineering]] — 设计信息环境
- 2026：[[Harness Engineering]] — 构建形式化约束环境

方向始终是：**从自然语言走向形式化约束**。

### Dijkstra 的窄接口原则

[[Edsger Dijkstra]] 的"窄接口"（narrow interface）原则认为：约束越明确，合作越高效。让接口变宽（如自然语言），跨接口的沟通成本反而会增加。CLAUDE.md、AGENTS.md、TDD 测试套件、CI/CD 管道——这些都是 AI 时代"窄接口"的具体形态。

> 测试和约束是排除 AI 胡说八道的工具。

### 生产环境实践：叶子节点策略与可验证抽象层

[[Erik Schluntz]]（Anthropic 研究员）在真实生产环境实践中提出了两个关键策略：

**叶子节点（Leaf Nodes）策略**

将 Vibe Coding 限制在代码库中不被任何其他模块依赖的末端功能：

- **叶子节点区域**：末端功能、附加组件，极少变动，产生技术债可接受
- **主干区域**：仍需人工深入理解，保护底层可扩展性
- **动态边界**：随模型能力提升，可信任 AI 接管的层级在向下延伸

**可验证抽象层（Verifiable Abstraction Layer）**

开发者需找到“无需阅读底层代码即可验证正确性”的抽象层，类比企业管理：

| 角色 | 验证方式                    |
| ---- | --------------------------- |
| CTO  | 验收测试（acceptance test） |
| PM   | 体验产品                    |
| CEO  | 关键数据切片                |

TDD 测试套件是工程师建立可验证抽象层的核心手段。

**Anthropic 内部实战案例**：22000 行代码合并，原本需要两周的工程压缩到 1 天，四大策略：PM 视角深度引导 + 叶子节点限制 + 核心区域人工介入 + 可验证检查点（长时间压力测试）。

### TDD 在 Vibe Coding 中的应用

> 「即使你看不懂测试用例，它也能帮助 Claude 变得更加自洽。」——[[Erik Schluntz]]

实践原则：

- **极简 E2E 测试**：强制规范「只写3 个测试：happy path + 错误场景 1 + 错误场景 2」
- **避免过度测试**：引导 Claude 写端到端测试而非过度依赖具体实现的单元测试
- **测试即唯一审查点**：Vibe Coding 时唯一需要审查的代码就是测试代码
