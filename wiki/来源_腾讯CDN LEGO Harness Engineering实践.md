# 腾讯CDN LEGO Harness Engineering 实践

## 基本信息

- **来源**：微信公众号"腾讯程序员"
- **作者**：腾讯CDN LEGO 项目团队
- **副标题**：AI 大规模系统实现可交付代码全指南
- **关键问题**：AI 能在真正"出事会炸"的后端系统（百万行、亿级用户）里可靠地写代码吗？

## 核心要点

1. **规模与挑战**：腾讯 CDN LEGO 系统核心代码超 100 万行，理论组合路径高达 13,824×N 种，是真正的高风险后端系统
2. **能力验证**：20 天 1 人 + AI 零人工代码完成 Rust 版 nonstop 代理框架（42,052 QPS / 5000 并发 0 错误）
3. **问题归因**：基于 57 个真实案例提炼 13 类典型问题 + 5 大根因，根本原因是 AI 缺乏"不确定性意识"和"全局视野"
4. **五层架构**：围绕"上下文、约束、反馈"三大要素构建 LEGO Harness Engineering 五层架构
5. **对抗式CR**：三模型（Claude + Codex + Gemini）并行独立审查 + 交叉迭代 + 自动收敛，解决单模型的三大盲区
6. **工程是核心资产**：工程体系才是持续积累的价值，Skill/模型每天更新，但工程经验持续沉淀

## 详细摘要

### 背景：为什么要在高风险后端系统里引入 AI Coding

LEGO 系统的挑战清单：
- 核心代码超 100 万行，多线程全异步非阻塞架构
- 深度改造第三方库超 300 万行（OpenSSL、QUIC、LUA、JavaScript 等）
- 每天万亿级请求，服务亿级用户
- 面对不可控客户端 + 不可控源站，理论组合路径高达 13,824×N 种
- 异步编程复杂：Future/Promise 链路长，Lambda 生命周期管理易出错
- 容错度极低：流式处理，错误直接暴露给用户

团队的结论：不是"用 AI"，而是**"驾驭 AI"**——这是 Harness Engineering 的起点。

### nonstop 项目：探测 AI 编码能力边界

20 天，1 人 + AI 团队，Rust 零人工代码开发 nonstop 代理框架：

- 支持 L4/L7 代理、HTTP/3 QUIC、内置 WAF 纵深防御、V8 JS Workers 边缘计算
- 单二进制部署，零停机热加载
- 实测：42,052 QPS / 5000 并发 0 错误 / P50 延迟 1.1ms / 6 层纵深防御

**惊喜与疑问**：AI 能力确实强，但在 LEGO 这样百万行级、高可靠的 C++ 系统里"敢用"吗？

### AI Coding 的常见问题：13 类问题 + 5 大根因

基于 57 个真实案例提炼的问题清单（严重程度从高到低）：

| 问题类型 | 严重程度 |
| --- | --- |
| 不会说"我不知道"（用自信语气输出错误结论） | **最高** |
| 幻觉（编造 API、RFC 章节号、数据） | Critical/High |
| 异步语义误用（blocking send in tokio） | Critical |
| 安全盲区（时序攻击/SSRF/JWT） | Critical |
| 内存泄漏（DashMap 只增不减） | Critical |
| 协议实现不完整 | Critical |
| 改不全（insert 无 cleanup） | High |
| 底层未读就改上层 | High |
| 源码分析替代实测 | High |

**5 大根因**：不确定性意识缺失、幻觉、局部修改遗漏全局影响、模式匹配代替验证、缺乏环境意识

> 根本原因：AI 缺乏"不确定性意识"和"全局视野"。

### LEGO Harness Engineering 五层架构

核心理念：将 AI 尽量 harness 在**单个模块、单个文件、单个函数**内实现。三大核心要素：**上下文、约束、反馈**。

**上下文建设**（消除 AI 的"记忆偏差"）：
- 四层递进体系：Agent.md（项目宪法）→ 安全纪律（反例免疫）→ 可复用模式库 → 专业 Skill
- 竞品调研 Agent 团队：技术决策三问（RFC 怎么说？业界怎么做？LEGO 有什么差别？）
- RFC 原文本地化：将 38,068 行 RFC 固化在本地，AI 直接读取而非"回忆"

**约束**（用结构化约束替代语言化期望）：

> 期望"写高质量的代码" → AI 理解模糊，输出不稳定
> 约束"禁止裸 new，必须 unique_ptr" → AI 100% 遵循

三层约束：权限安全基座 + 代码规则编译器 + 流程约束（测试不可跳过）

五条核心约束均来自真实踩坑，例如：
- 本地不存在则跳过（防止 AI 编造"源码分析"）
- 严禁网络操作（防止引用过时竞品信息）

**反馈**（反馈速度决定进化速度）：
- 9 阶段全自动流水线（一条命令驱动）
- 三条并行反馈通道：自动采集（Hook）+ 踩坑日志 + CLAUDE.md 内联反馈
- 进化闭环：真实踩坑 → 安全规则 → CR 检查清单 → A/B 实验验证

### [[对抗式CR]]：多模型多 Agent 交叉审查

单模型 CR 的三个盲区：
- **知识盲区**：每个模型训练数据不同
- **注意力盲区**：500+ 行 diff 时后半部分审查质量下降
- **确认偏差**：发现一类问题后倾向沿同方向继续找

对抗式 CR：Claude + Codex + Gemini 三模型**并行独立审查** → 交叉验证 → 辩论式讨论（同意/反对/维持）→ 全员无新发现时**自动收敛**

### 阶段性收益

综合效率提升 20%（含 AI 执行结果人工 Review + 新人学习曲线）

各环节提升：
- 竞品调研：3 人天 → 1 天（约 3x）
- 协议安全测试：3-5 人天 → 1 天（约 4x）
- 代码审查：等待 1-3 天 → 30 分钟
- cpplint 通过率：>95%

知识资产积累：86,422 行代码、31 个 Skill、34 条踩坑规则

### 剩余挑战

- 误报率 36%（9 个问题中真实 P0 仅 1 个）
- 文档爆炸：8 个需求生成 99 个文件，人难以全部确认
- AI 的"自信"会传染：格式工整的文档反而降低审查意愿
- 团队能力退化风险：AI 用多了，工程师专业和文档能力可能下滑

### 工程师角色演变

- 初级开发 → AI 操作员（掌握 Skill 与 Prompt）
- 高级开发 → Harness 工程师（设计约束、上下文与规则）
- 架构师 → 人机协作架构设计（判断什么交给 AI、什么人来把控）
- 测试/安全工程师 → AI 质量工程师 / AI 安全专家

> 不变的核心能力：**抽象思维**——知道什么该让 AI 做、如何验证 AI 做得对不对。

## 关键引用

> 工程体系才是核心资产，而不是某个模型或 prompt。Skill 每天在更新，大模型在进化，但工程体系的价值持续积累。

> AI Coding 不是"让 AI 替你写代码"，而是重新定义人与 AI 协作的工程范式。每一个踩坑变成规则，每一条规则内化进 Skill，每一个 Skill 让下一个人少走弯路——这是一套可持续进化的工程体系。
