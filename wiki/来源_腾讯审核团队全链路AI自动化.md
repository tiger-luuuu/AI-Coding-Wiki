# 腾讯审核团队：从提需求到部署发布的全AI自动化实践

## 基本信息

- **来源**：微信公众号"腾讯程序员"
- **作者**：腾讯审核团队
- **核心问题**：如何将 AI 编码的价值从单一 Coding 环节扩展到需求→设计→开发→测试→部署的完整链路？

## 核心要点

1. **三阶段演进**：L1（纯人工）→ L2（人机协同）→ L3（AI 全自动），目前处于 L2，目标 L3
2. **四大核心挑战**：交付流程标准化、需求描述规范化、高质量知识库搭建、核心 Skills 标准化
3. **公式提炼**：AI 全自动化交付 = LLM + Harness Engineering（工作流程 + 知识库 + Skills） + Business Engineering
4. **效果数据**：代码行采纳率 90%+，AI 生成率 80%+，需求/技术方案评分 80 分以上
5. **双框架体系**："交付框架"（效率）+ "治理框架"（质量）并行运转

## 详细摘要

### 现状与目标

团队认为当前的主要问题不在编码本身，而在于：

> 从需求到交付的全链路中，大量人工仍耗费在**方案设计、代码评审、测试验证**等非编码环节，频繁的上下文切换与协作等待严重制约了研发效能。

演进目标：整体研发效能提升 **80%**。

### 三阶段演进路线

| 阶段 | 描述 | 核心工作 |
| --- | --- | --- |
| L1 | 纯人工 | 所有环节由人完成 |
| **L2**（当前） | 人机协同 | 以"技术方案生成代码"为核心，打通测试、部署工具链 |
| L3（目标） | AI 全自动 | 需求到部署端到端无人值守交付 |

L3 的三段式能力阶梯：技术方案→代码 → 技术方案→交付 → 需求→交付（端到端跑通）

### 四大核心挑战与解法

| 挑战 | 关键举措 |
| --- | --- |
| 交付流程标准化 | 流程编排 Skill，标准化研发流程，支持出错自动回滚 |
| 需求的标准化 | 标准需求模板 + 澄清规则 + 评分规则 + PRD-Agent |
| Skills 的沉淀 | 技术方案生成、代码生成、代码 CR、测试用例执行等标准 Skill |
| 知识库搭建 | 业务知识库 + 代码知识库，为 AI 提供高质量上下文 |

### L2 人机协同的具体实现

**技术方案到代码**（第一年重点）：
- AI 技术规范（索引/rules）+ 模板化技术方案/prompt + MCP 工具集成 + AI 自我总结
- 对应[[胶水编程]]的核心思路：上下文决定上限

**技术方案到交付**（当前扩展）：
- 扩展技术方案为执行清单（研发清单 + 测试清单 + 发布清单）
- 研发阶段：分层生成代码 + 单元测试覆盖
- 测试阶段：通过 MCP 工具（七彩石、DDL/DML 工单、蓝盾流水线）一键创建测试环境
- 部署阶段：接入无人值守，自动执行发布单

**架构收益**：
- 跨平台减少：从 6 个平台 → 在 CodeBuddy 内闭环
- 步骤减少：12 步 → 5 步（理想 2 步）
- 测试阶段节约时长 1h（60%+），部署阶段节约 0.5 天（50%）

**需求到代码**（最新阶段）：
沉淀了 1 个 PRD-Agent、5 套规范化标准、3 个知识库、3 项核心技能。

### L3 全自动化架构

引入 [[Harness Engineering]] 理念后构建全流程自动化框架：

```
需求 → 方案 → 开发 → 测试 → 发布
  ↓      ↓      ↓      ↓      ↓
需求评审 方案评审 质量门禁  MR卡点  闭环反馈
```

特点：工程卡点保障质量 + 闭环反馈与卡点自动修复 + 多 AI Agent 自主协同

**效果数据（6个试点需求，三轮迭代）**：
- 需求评分：稳定在 80 分以上
- 技术方案评分：80 分以上
- 平均生成对话轮次：2 轮
- 代码行采纳率：90% 以上
- AI 生成率：80% 以上

### 未来展望：双轮驱动体系

**AI 全自动化交付框架**：需求→设计→开发→测试→发布的全流程自动化 + 内置工程卡点 + 闭环反馈与自修复

**AI 全自动化治理框架**：运行时日志 + 监测指标 + Spec 知识 + 中间产物 → 线上服务自动巡检与修复 + 异常实时感知 + 知识 Skill 自动回溯更新

> **"交付驱动效率，治理保障质量"** 的双轮驱动体系
